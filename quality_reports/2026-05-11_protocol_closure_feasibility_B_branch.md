# Protocol Closure: Feasibility and the B Branch

Date: 2026-05-11

Status: pending protocol decision after independent verification reservations.

## Scope

This report records the protocol interpretation used to support the
high-state-only branch \(B\) in the \(\pi_H=0\) baseline. It is not a new
behavioral assumption hidden inside a proof. It is an explicit closure of the
public history generated by infeasible proposals.

## Primitive Feasibility Object

In the unanimity R1 branch, a weak proposer promises \(h\) to \(H\) and \(y\) to
each of the \(k=m-1\) non-proposing weak voters. Let

\[
T=h+ky.
\]

The proposal is executable in realized state \(\theta\) only if
\(T\leq V(\theta)\). Since \(V(0)=1\) and \(V(1)=r>1\), a proposal with

\[
1<T\leq r
\]

is high-state feasible and low-state infeasible.

The protocol makes this feasibility primitive operational through a
nonstrategic implementation check. The check is not \(H\), not the proposer, and
not another strategic player. It has no discretion. It publicly reports whether
the promised total can be executed against the realized pie:

- \(F\): feasibility failure, \(T>V(\theta)\);
- \(E\): executable proposal, \(T\leq V(\theta)\).

No strategic actor can misreport feasibility.

The implementation check is run only after all required voters have consented.
If any required voter withholds consent, the proposal fails by consent, no
feasibility check is run, and the game proceeds to R2 after a generic
no-consent history.

## Protocol Closure

Mechanical feasibility failure means that a proposal that does not fit the
realized pie fails mechanically at the implementation check. The public history
records "feasibility failure," not an individual vote or rejection by \(H\).

Therefore, under \(1<T\leq r\), after all required voters consent:

- in the low state, the proposal fails mechanically, the posterior becomes
  zero, and the game moves to the low-posterior R2 continuation;
- in the high state, the proposal is feasible and passes if \(H\) and the
  non-proposing weak voters accept.

No off-path belief about \(H\)'s low-state vote is needed because no individual
\(H\) vote is publicly recorded on the low-state failure path.

The failed R1 proposal consumes the first bargaining round. There is no
revision, resubmission, or redraw of an R1 proposer after feasibility failure.
The next node is Round 2 with posterior \(\mu'=0\) and the same baseline
recognition rule \(\pi_H=0\).

The timing is:

1. a weak proposer announces \(T=h+ky\);
2. required voters choose whether to consent;
3. if any required voter withholds consent, no feasibility check is run, the R1
   proposal fails by consent, and the game proceeds to R2 after a generic
   no-consent history;
4. if all required voters consent, the nonstrategic implementation check
   determines whether \(T\leq V(\theta)\);
5. if \(T>V(\theta)\), the only public event is \(F\), the R1 proposal fails,
   and the game proceeds to R2;
6. if \(T\leq V(\theta)\), the agreement is implemented.

On the equilibrium \(B\) path, event \(F\) occurs only in the low state and
implemented agreement occurs only in the high state. Individual votes are not
separately reported on the \(F\) history.

For the off-path generic no-consent history, no feasibility check is run and no
individual \(H\) vote is reported. The continuation belief is \(\mu\). This
belief is supported by a sequence of state-independent weak-voter trembles:
if generic no-consent is generated by weak voters with probabilities that do not
depend on \(\theta\), Bayes' rule leaves the posterior unchanged. This
no-learning belief is used only for the deviation payoff in the weak-voter IC.
It is distinct from the on-path feasibility-failure history \(F\), where
posterior zero follows directly from \(1<T\leq r\).

## Sequential Rationality Conditions

The high type of \(H\) accepts when

\[
h_B=\beta\alpha r.
\]

This uses the maintained tie-breaking convention that players accept at
indifference.

An uninformed weak voter accepts when

\[
\mu y+(1-\mu)\beta W_2^U(0)\geq \beta W_2^U(\mu).
\]

Thus, for \(\mu>0\),

\[
y_B(\mu)
=
\frac{\beta W_2^U(\mu)-(1-\mu)\beta W_2^U(0)}{\mu}
=
\frac{\beta\{g(\mu)-(1-\mu)(1-\alpha)\}}{m\mu}.
\]

The total promised transfer is

\[
T_B(\mu)
=
\beta\alpha r
+
(m-1)\frac{\beta\{g(\mu)-(1-\mu)(1-\alpha)\}}{m\mu}.
\]

The branch has an exact maximum when

\[
1<T_B(\mu)\leq r.
\]

On that region,

\[
B(\mu)
=
\mu\{r-T_B(\mu)\}
+
(1-\mu)\frac{\beta(1-\alpha)}{m}.
\]

The weak proposer chooses \(B\) only when this feasibility condition holds and
\(B(\mu)\) weakly dominates both the accepted-pooling branch \(P(\mu)\) and the
no-information rejection branch \(R(\mu)\).

## Caveat

If the extensive form publicly records an individual \(H\) vote before or
independently of the feasibility check, the analysis changes. That protocol
would require explicit off-path beliefs after unexpected \(H\) voting histories
and a new verification pass. The current \(B\) branch should not be used under
that alternative protocol without approval.

The same is true if the feasibility check is run even after a voter withholds
consent. In that alternative timing, no-consent on a high-state-only offer would
also reveal whether \(T\) fits the realized pie, so the weak-voter IC must be
rederived.

2026-05-11, with only locale startup warnings.

`model_redesign/power_architecture_derivations.Rmd` compiled successfully to

The first independent verification on 2026-05-11 returned PASS without
reservations. After the user asked who verifies feasibility and where the game
moves after failure, the Rmd section and this report were clarified to specify
a nonstrategic implementation check and R2 continuation after failure.

A second independent verifier returned FAIL/reservation because the previous
timing could be read as running the feasibility check even after a voter rejects,
which would make the no-consent history state-revealing and invalidate the
weak-voter IC.

A third independent verifier returned FAIL/reservation after the text was
revised to make consent precede implementation. The remaining problem is that,
under unanimity, \(H\) is a required voter. If low-type \(H\) can withhold
consent before the feasibility check, the low type has a consent IC that
generally fails in the calibrated \(B\) region. Low \(H\) would rather withhold
consent and continue at the generic no-consent belief than consent to a proposal
that mechanically fails and reveals posterior zero.

The protocol closure is therefore not verified. The next modeling decision must
choose explicitly between at least two protocols:

1. only non-proposing weak voters consent before the implementation check, and
   \(H\) is asked to accept only after the check reports executability;
2. \(H\) consents before implementation, in which case the \(B\) branch must add
   low-type \(H\)'s consent IC and the current calibrated high-\(\mu\) \(B\)
   region is not verified.
